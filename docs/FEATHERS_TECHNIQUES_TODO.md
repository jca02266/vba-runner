# Michael Feathers "Working Effectively with Legacy Code" Techniques TODO（レガシーコード改善ガイド 手法リスト）

## このドキュメントについて

Michael Feathers の著書 *"Working Effectively with Legacy Code"*（邦題：レガシーコード改善ガイド）は、
テストのないレガシーコードを安全にリファクタリングするための**オブジェクト指向的手法のカタログ**として広く参照されている。

このファイルは同書で紹介されているリファクタリング手法・テスト手法・設計パターンを一覧化したものであり、
このプロジェクトのリファクタリング支援を設計・拡張する際の参考文献として位置づける。

各手法のうち、このプロジェクトのカタログ（`docs/REFACTORING_TESTING_CATALOG.md`）に収録済みのものに `[x]` を付けている。
VBA はオブジェクト指向の制約（継承なし・コンストラクター引数なし）があるため、
関数抽出（Extract Method）系（Seam, Sprout Method, Wrap Method）が最も現実的な適用手法になる。

## Refactoring Techniques（リファクタリング手法）

| 手法 | 概要 | カタログ |
|---|---|---|
| [x] **Seam（縫い目）** | テスト可能にするための差し替えポイントを作る。副作用を関数抽出してモック可能にする | [R-13] |
| [x] **Sprout Method（芽生えメソッド）** | 既存コードを変えず、新機能を別メソッドに実装して呼び出す。変更箇所を最小化する | [R-17] |
| [ ] **Sprout Class（芽生えクラス）** | Sprout Method の拡張。新機能のためにクラスを新たに作成して既存コードから呼ぶ | — |
| [x] **Wrap Method（ラップメソッド）** | 既存メソッドを別名にリネームし、同名の新メソッドで旧実装を呼び出す形にして前後処理を追加する | [R-18] |
| [ ] **Wrap Class（ラップクラス）** | Decorator パターン。既存クラスをラップして振る舞いを追加する。既存コードを一切変更しない | — |
| [ ] **Extract and Override（抽出とオーバーライド）** | 変更したい部分を protected メソッドに抽出し、テスト用サブクラスでオーバーライドして差し替える | — |
| [ ] **Introduce Instance Delegator（インスタンスデリゲーターの導入）** | static メソッドをインスタンスメソッドに変換し、テストでインスタンスを差し替え可能にする | — |
| [ ] **Break Out Method Object（メソッドオブジェクトへの切り出し）** | 大きなメソッドをクラスに変換。ローカル変数がフィールドになり、内部メソッドへの分割が容易になる | — |
| [ ] **Expose Static（スタティック公開）** | グローバル・シングルトン依存をパラメーター経由にして差し替え可能にする | — |
| [ ] **Parameterize Constructor（コンストラクターのパラメーター化）** | 依存オブジェクトをコンストラクターに注入できるようにして、テストで差し替えを可能にする | — |
| [ ] **Parameterize Method（メソッドのパラメーター化）** | メソッド内でハードコードされた依存をパラメーターとして受け取れるようにする | — |
| [ ] **Pull Up Feature（フィーチャーの引き上げ）** | テストしたいコードをスーパークラスに引き上げてテスト可能なサブクラスから呼び出す | — |
| [ ] **Subclass and Override（サブクラス化とオーバーライド）** | テスト用にサブクラスを作り、依存の強い部分をオーバーライドで無効化する | — |
| [ ] **Supersede Instance Variable（インスタンス変数の置き換え）** | インスタンス変数を後から差し替えるための setter を追加してテスト可能にする | — |

## Testing Techniques（テスト手法）

| 手法 | 概要 | カタログ |
|---|---|---|
| [x] **Characterization Test（特性テスト）** | 現在の挙動をそのまま記録する。仕様書への準拠ではなく「今のコードが何をしているか」のベースライン確保 | [T-14] |
| [x] **Snapshot Test（スナップショットテスト）** | 戻り値または副作用呼び出し列を記録し、リファクタリング後の変化を検出する | [T-13] |
| [ ] **Sensing Variable（検知変数）** | 副作用を確認するためにテスト専用の変数を追加して状態を外部から観察できるようにする | — |
| [ ] **Link Seam（リンク縫い目）** | ビルド時にリンクするモジュールを差し替えることで依存を切り離す（C/C++ 向けだが概念として参照） | — |
| [x] **Object Seam（オブジェクト縫い目）** | ポリモーフィズムを利用して依存オブジェクトを差し替える縫い目。VBA では Implements で実現 | [T-15] |
| [ ] **Preprocessing Seam（プリプロセッサー縫い目）** | コンパイル時にコードを差し替える縫い目（マクロ定義など）| — |
| [ ] **Effect Sketch（影響スケッチ）** | 変更箇所から波及する影響をグラフ（スケッチ）として書き出し、テストすべき箇所を特定する | — |
| [ ] **Pinch Point（ピンチポイント）** | 多くの影響が通過する狭い箇所を特定し、そこにテストを置くことで広い範囲をカバーする | — |
| [ ] **Interception Point（傍受ポイント）** | テストが観察できる最初のポイント。縫い目との組み合わせで使う | — |

## Related Design Patterns（関連設計パターン）

| パターン | 概要 | カタログ |
|---|---|---|
| [x] **Functional Core / Imperative Shell - FC/IS** | 副作用が一方向フローの場合に Core と Shell を分離する設計 | [R-12] |
| [ ] **Dependency Injection（依存性注入）** | 依存オブジェクトを外部から渡すことでテスト時に差し替え可能にする設計原則 | — |
| [ ] **Humble Object（謙虚なオブジェクト）** | テストしにくい環境依存のコードを最小化し、テスト可能なロジックを別オブジェクトに移す | — |
| [x] **Command/Query Separation - CQS（コマンド・クエリー分離）** | 状態を変えるメソッド（Command）と値を返すメソッド（Query）を明確に分けることで副作用を局所化する | [R-01] |
