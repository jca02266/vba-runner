## 次のステップ：テーブル設計の自動提案

### レベル 1: 単純なテーブル構造提案

**入力**: 検出された分岐パターン

**出力**: Type 定義と初期化テンプレート

```typescript
// 検出情報から Type 定義を自動生成
function proposeTableStructure(candidate: TableDrivenCandidate): string {
    const outerKey = "department";        // 外側分岐のキー名
    const outerValues = ["Sales", "Marketing", "IT", "HR", "Finance"];  // 検出値
    
    const innerConditions = [             // 内側分岐の条件値
        { variable: "amount", values: [50000, 500000, 2000000] }
    ];
    
    const assignmentResult = "approver";  // 結果変数名
    
    // Type 定義を生成
    return `Type ApprovalRule
    ${outerKey} As String
    ${innerConditions.map((c, i) => 
        `threshold${i+1} As Long
         ${assignmentResult}${i+1} As String`
    ).join('\n    ')}
    default${assignmentResult} As String
End Type`;
}
```

**挙動**:
1. 外側分岐の条件値（="Sales", "Marketing", ...）を抽出
2. 内側分岐の条件値（amount < 50000, < 500000, ...）を抽出
3. 代入結果のデータ型を推測（String, Long, Double など）
4. Type 定義テンプレートを生成

**実装難度**: ⭐⭐（中程度）

**実装に必要な拡張**:
- AST から条件値を抽出するロジック
- リテラル値の型推定
- Type 定義テンプレート生成エンジン

### レベル 2: テーブルデータの自動抽出

**入力**: 分岐パターンの詳細分析結果

**出力**: テーブルデータ（JSON/CSV）

```typescript
// 分岐内容をテーブル形式で構造化
interface TableRow {
    department: string;
    threshold1: number;
    approver1: string;
    threshold2: number;
    approver2: string;
    // ...
}

function extractTableData(branches: IfStatement[]): TableRow[] {
    // 各分岐から以下を抽出：
    // 1. 外側分岐の条件値（department = "Sales"）
    // 2. 内側分岐の条件値と結果（amount < 50000 → "Manager"）
    
    return branches.map(branch => {
        const dept = extractConditionValue(branch.condition);  // "Sales"
        const rules = extractInnerRules(branch.consequent);     // [{threshold: 50000, approver: "Manager"}, ...]
        
        return {
            department: dept,
            threshold1: rules[0].threshold,
            approver1: rules[0].approver,
            threshold2: rules[1].threshold,
            approver2: rules[1].approver,
            // ...
        };
    });
}
```

**出力例**:
```csv

Sales,50000,Manager,500000,Director,2000000,VP,CFO
Marketing,30000,Manager,300000,Director,1500000,VP,CFO
IT,100000,Manager,800000,Director,3000000,VP,CFO
HR,20000,Manager,200000,Director,1000000,VP,CFO
Finance,10000,Manager,100000,Director,500000,VP,CFO
```

**利点**:
- テーブルデータを Excel や Database に直接インポート可能
- ビジネスアナリストによる検証・修正が容易
- 将来的に外部データソースへの移行が容易

**実装難度**: ⭐⭐⭐（難）

**実装に必要な拡張**:
- 代入ステートメントの詳細解析
- リテラル値の抽出と型変換
- 順序の保証（AST トラバーサルの複雑性）

### レベル 3: テストケースの自動生成

**入力**: テーブルデータ + 元の分岐構造

**出力**: VBA テストコード

```typescript
// テストケースを自動生成：カバレッジ = 外側分岐数 × (内側分岐数 + 1 [デフォルト])
function generateTestCases(tableData: TableRow[], candidate: TableDrivenCandidate): string {
    let testCode = `Sub TestApprovalRulesAutoGenerated()\n`;
    testCode += `    InitializeApprovalRules\n\n`;
    
    for (const row of tableData) {
        // 各分岐について (内側分岐数 + 1) 個のテストケースを生成
        const department = row.department;
        
        // threshold1 未満 → approver1
        testCode += `    ' ${department}部, threshold1未満\n`;
        testCode += `    Debug.Assert GetApprover(${row.threshold1 - 1}, "${department}") = "${row.approver1}"\n`;
        
        // threshold1 以上 threshold2 未満 → approver2
        testCode += `    ' ${department}部, threshold1以上threshold2未満\n`;
        testCode += `    Debug.Assert GetApprover(${row.threshold1 + 1000}, "${department}") = "${row.approver2}"\n`;
        
        // threshold2 以上 threshold3 未満 → approver3
        testCode += `    ' ${department}部, threshold2以上threshold3未満\n`;
        testCode += `    Debug.Assert GetApprover(${row.threshold2 + 1000}, "${department}") = "${row.approver3}"\n`;
        
        // threshold3 以上 → defaultApprover
        testCode += `    ' ${department}部, threshold3以上\n`;
        testCode += `    Debug.Assert GetApprover(${row.threshold3 + 1000000}, "${department}") = "${row.defaultApprover}"\n\n`;
    }
    
    testCode += `    MsgBox "All tests passed!"\n`;
    testCode += `End Sub\n`;
    
    return testCode;
}
```

**生成されるテスト**:
```vba
Sub TestApprovalRulesAutoGenerated()
    InitializeApprovalRules

    ' Sales部, threshold1未満
    Debug.Assert GetApprover(49999, "Sales") = "Manager"
    ' Sales部, threshold1以上threshold2未満
    Debug.Assert GetApprover(501999, "Sales") = "Director"
    ' Sales部, threshold2以上threshold3未満
    Debug.Assert GetApprover(1001000, "Sales") = "VP"
    ' Sales部, threshold3以上
    Debug.Assert GetApprover(3001000, "Sales") = "CFO"

    ' Marketing部, threshold1未満
    Debug.Assert GetApprover(29999, "Marketing") = "Manager"
    ' ... 全部門カバレッジ自動生成

    MsgBox "All tests passed!"
End Sub
```

**テストカバレッジ**: `部門数 × (閾値数 + 1)` = 5 × 4 = 20 テストケース

**実装難度**: ⭐⭐（中程度）

**実装に必要な拡張**:
- テーブルデータから VBA コード生成
- 条件値の数値演算（境界値テストのための ±1 計算）
- Debug.Assert 生成ロジック

### レベル 4: 双方向検証テスト

**課題**: リファクタリング前後の動作が同一か保証したい

**入力**: 元のコード + リファクタリング後のコード

**出力**: 両者を比較検証するテストコード

```typescript
// リファクタリング前後の結果を比較
function generateComparisonTest(
    originalFunc: string,  // GetApprover（元のコード）
    refactoredFunc: string, // GetApprover（テーブル版）
    testCases: TestCase[]
): string {
    let testCode = `Sub TestRefactoringEquivalence()\n`;
    
    for (const tc of testCases) {
        const originalResult = `GetApprover_Original(${tc.amount}, "${tc.department}")`;
        const refactoredResult = `GetApprover_Refactored(${tc.amount}, "${tc.department}")`;
        
        testCode += `    Debug.Assert ${originalResult} = ${refactoredResult}, `;
        testCode += `"Mismatch at (${tc.amount}, ${tc.department})"\n`;
    }
    
    testCode += `    MsgBox "All equivalence tests passed!"\n`;
    testCode += `End Sub\n`;
    
    return testCode;
}
```

**利点**:
- リファクタリング後の回帰バグを即座に検出
- 自信を持ってコードを置き換え可能
- テスト駆動リファクタリング（TDD）が実現可能

**実装難度**: ⭐⭐⭐⭐（非常に難）

**実装に必要な拡張**:
- 元のコードの Function を別名で保持
- 両関数の同時実行と結果比較
- 網羅的なテストケース生成アルゴリズム

## 技術的な課題と解決策

### 課題 1: 複雑な条件式の分解

**問題**:
```vba
' 単純な比較
If amount < 50000 Then  ' ← 抽出可能：{variable: "amount", operator: "<", value: 50000}

' 複雑な論理式
If amount < 50000 AND region = "Tokyo" Then  ' ← 抽出困難
    If salesman = "A" Then  ' ← 3次元テーブルになる
        result = "Special"
    End If
End If
```

**解決策**:
- 単純なバイナリ比較（`<`, `>`, `=`）のみサポート（Phase 1）
- 複雑な論理式は検出時に"非推奨"フラグで除外
- 将来的に Strategy Pattern への自動リファクタリングを提案

### 課題 2: テーブル行の構造化

**問題**: 内側分岐の数が異なる場合

```vba
If department = "Sales" Then
    If amount < 50000 Then
        result = "Manager"
    ElseIf amount < 500000 Then
        result = "Director"
    End If  ' ← 2閾値のみ
ElseIf department = "Marketing" Then
    If amount < 30000 Then
        result = "Manager"
    ElseIf amount < 300000 Then
        result = "Director"
    ElseIf amount < 1500000 Then
        result = "VP"
    End If  ' ← 3閾値
End If
```

**解決策**:
- 内側分岐数の最大値に統一（不足分は NULL または Variant.Empty）
- 非対称テーブルの場合は検出時に警告を発行

### 課題 3: テストケース数の爆発

**問題**: テストカバレッジが多い場合

```
5部門 × 4閾値 = 20 テストケース
10部門 × 5閾値 = 50 テストケース
```

**解決策**:
- 代表的な境界値テストのみを生成
- 全組み合わせのカバレッジオプションは別途提供
- パラメータ化テスト（`@DataProvider`）パターンで効率化
