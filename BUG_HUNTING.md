# BUG_HUNTING.md — バグ発見手法とツールの使い方

VBA 実行エンジンのバグを体系的に発見するための手法カタログ。
それぞれ狙うバグの種類が異なり、相互補完の関係にある。
発見したバグの修正フローは通常どおり（`tests/spec/` にテスト追加 → 修正 → `TODO_SPEC.md` 記録 → コミット → `bg-test.sh`）。

## バグの分類と手法の対応

| バグの種類 | 例 | 有効な手法 |
|---|---|---|
| クラッシュ（JS 生例外） | `Number(Symbol)` で TypeError | ①ファザー ②On Error ストレス |
| サイレントな値の誤り | `Tab(10)` が 11 桁目に出力 | ③メタモルフィック ④実 VBA 差分 ⑧評価ループ |
| 黙った機能欠落 | `Erl` が未実装で暗黙 0 | ⑧評価ループ ④実 VBA 差分 |
| 整形・LSP の破壊 | フォーマッターがコードを壊す | ⑦往復テスト |
| テスト自体の穴 | 何も検証していないテスト | ⑥ミューテーション ⑤カバレッジ |

---

## 実施済みの手法（常設ツール）

すべて**テストスイート外のオンデマンドツール**。毎回のリグレッションテスト（`bg-test.sh`）の実行時間には影響しない。
ツールで見つけたバグは修正後、代表ケースだけを `tests/spec/` の軽いリグレッションテストとしてスイートに固定する方式。

### ① 組み込み関数の異常値スモークファザー

```bash
npx tsx scripts/fuzz-builtins.ts [関数名...]   # 引数なしで全関数。バグ候補ありなら exit 1
```

- 登録済みの全組み込み関数（184 名、`$` 変種含む）× 敵対値 12 種（`Null` / `Empty` / `Nothing` / `""` / 巨大数 / 配列 等）を各引数位置に総当たり注入
- **VBA エラーで落ちるのは正常**とみなし、JS 生例外（TypeError / RangeError 等）で落ちるものだけを検出する
- 実績: 導入時 359 件検出 → Bug DT として全修正（0 件到達）
- **再実行タイミング**: `builtins.ts` / `coerce.ts` / evaluator の組み込み関数まわりを変更したとき
- 関連規約: 引数の型強制は生 `Number()` / `String()` ではなく中央ヘルパー（`ctx.toVbaNumber` / `vbaToString`）を使う。Nothing→Error 91、Null→13/94、配列→13 が一括で効く

### ② On Error 経路の組み合わせストレステスト

```bash
npx tsx scripts/stress-on-error.ts   # 決定的（乱数なし）。不一致ありなら exit 1
```

- ハンドラーモード（5 種）× エラー発生位置 × エラー種別（Err.Raise / 0 除算 / 型不一致 / 添字）× 構造（インライン / For ループ内 / 子プロシージャ / 二重エラー / ハンドラー内エラー）= 128 シナリオを機械生成
- `Debug.Print` の実行トレースを MS-VBAL §5.4.4 準拠の期待値モデルと突き合わせる
- 実績: 導入時に Bug DU 2 件発見（ネストブロック内 `Resume Next` のループ脱出、`Err.Raise` の既定 Description 欠落）。「`End Sub` は Err を自動クリアしない」という仕様裁定も獲得
- **再実行タイミング**: エラーハンドリング機構（`executeStatements` / `ErrObject` / Resume 処理）を変更したとき
- 注意: モデル側が仕様の理解を誤っている可能性もある。不一致 = 即バグではなく、MS-VBAL で裁定してからどちらを直すか決める

### ③ メタモルフィックテスト（恒等式ベース）

```bash
npx tsx scripts/metamorphic-test.ts [seed] [iterations]   # 既定 200 回/恒等式
npx tsx scripts/metamorphic-test.ts --selftest            # 検出器自体の健全性確認
```

- オラクル不要の VBA 恒等式 40 個（`Mid(s,1,Len(s))=s`、`(a\b)*b+(a Mod b)=a`、`Join(Split(s,d),d)=s`、Hex/Oct/Val 往復、Currency の正確な加法、DateAdd 逆演算、`Sin²+Cos²=1` 等）にシード付きランダム入力を流し、破れたら報告
- **「落ちないが値が間違っている」バグ**を狙う。①が原理的に見つけられない領域
- seed が出力されるため違反は完全再現可能
- 実績: 導入時点で約 17 万試行違反ゼロ（式レベルの値計算が堅い裏付け）
- **再実行タイミング**: 文字列・数値・日付関数や型強制を変更したとき。恒等式の追加も歓迎
- `--selftest` は「偽の恒等式が正しく違反として検出されるか」を確認する。テストが実は何も見ていない事故の防止

### ⑦ フォーマッター往復テスト

```bash
npx tsx scripts/roundtrip-format.ts [ファイル...]   # 引数なしで repo 内全 .bas/.cls
```

- リポジトリ内の実 VBA コーパス（`sample/` `tests/` `tools/` の 99 ファイル）に対して 4 段階検査:
  1. `format()` がクラッシュしない
  2. 整形結果が再パースできる（整形がコードを壊していない）
  3. 整形前後で AST が意味的に同一（位置情報除去 + 識別子は大文字小文字無視で比較）
  4. 2 回目の整形が no-op（冪等性）
- 識別子ケーシングの統一（変数 `helper` とクラス `Helper` が共存すると片方に寄る）は実 VBE と同じ挙動 = 仕様。比較器はこれを吸収済み
- 実績: 導入時点で 99 ファイル全通過
- **再実行タイミング**: `src/lsp/formatter.ts` / `src/engine/parser.ts` / `src/engine/lexer.ts` を変更したとき

### ⑤ カバレッジ計測（探索の指針）

```bash
./scripts/coverage.sh            # 全テスト実行 + 集計（30 分〜1 時間）
./scripts/coverage.sh --report   # 既存の coverage-v8/ から集計だけやり直す
```

- それ自体はバグを見つけない。**「テストが一度も通っていない分岐 = 監査もされていない挙動」を可視化**し、評価ループ（⑧）のドメイン選定やテスト追加の優先度付けに使う
- `NODE_V8_COVERAGE` を `run_all_tests.sh` の全子プロセスに継承させて計測する
- ⚠️ **c8 の複数プロセスマージ（`--merge-async` 含む）は tsx 子プロセス構成で大幅に過少報告する**（実測: evaluator.ts 実際 92% に対し c8 マージは 51%）。本スクリプトはプロセスごとに個別変換して行カバレッジのユニオンを自前で取ることで回避している
- 実測（2026-07-18、全 913 プロセスのユニオン）: `evaluator.ts` 92.1% / `parser.ts` 92.6% / `builtins.ts` 94.8% / `coerce.ts` 93.8% / `lexer.ts` 98.4%
- 特定した主な暗部（詳細は `EVAL_LOG.md` の未テスト領域）: 日付リテラルの 2 要素形式解釈、`&H`/`&O` 文字列の数値強制、`Lock`/`Unlock` 文、`Open` 文の Access/Lock 節、IRR/MIRR・IPmt/PPmt の一部経路、Decimal 除算経路、`Set obj.Prop` のチェーン経路

### ⑧ ドメイン評価ループ（evaluate-vba-runner スキル）

Claude Code のスキル `/evaluate-vba-runner`（`/loop /evaluate-vba-runner` で連続実行）。

- サブエージェントに毎回**未評価のドメイン**で実用的な VBA プログラムを書かせ、新規ユーザー視点の「生の使用感」からバグ・使い勝手の問題を報告させる
- 報告されたバグは**必ずメインセッションで再現確認**してから修正する（サブエージェントの報告を鵜呑みにしない）
- 評価履歴・注意事項・未探索領域は `EVAL_LOG.md` に蓄積
- 実績: 評価 33 回で 60 件超のバグを発見・修正（黙った機能欠落や複合条件のバグに特に強い）

### ④ 実 VBA との差分テスト

```bash
npx tsx scripts/gen-diff-corpus.ts    # コーパス生成（diff-vba/ に .bas 群 + manifest）
npx tsx scripts/run-diff-local.ts     # vba-runner 側の実行 → diff-vba/results-runner.txt
# --- Windows + Excel 側 ---
#   1. 空の .xlsm を用意し vba-extractor import book.xlsm diff-vba/ で全モジュール注入
#   2. Excel で開き DiffMain.RunAll を実行 → vba-diff-results.txt が生成される
# --- 突き合わせ ---
npx tsx scripts/diff-results.ts diff-vba/results-runner.txt vba-diff-results.txt   # --full で全件一覧
```

- **同一の .bas を両環境で実行**して出力を diff する方式（VBA に式文字列の Eval がないため、式はコードに焼き込む）。式の解釈に非対称が入らない
- 各式は `On Error Resume Next` で包み、**値だけでなく `Err.Number` の一致（エラーコード互換性）も比較**する
- 値は locale 非依存の `Canon` 形式（`TypeName` + `Str`/明示 Format/文字列 16 進エンコード）で正規化。日本語ロケールの Excel でも揺れない
- コーパスは決定的に生成（2337 式: 演算子×型行列 1775 / 変換境界 308 / 文字列 133 / 日付 53 / 数学 30 / 財務 10 / エラー 28）。生成時に vba-runner のパーサーで事前スクリーニングし、超越関数・財務は式側で桁丸め
- 意図的な差分（コードページ依存・タイムゾーン依存等）は `scripts/diff-allowlist.txt` で許容管理。診断結果はパターン分類して表示される（`--full` で全件一覧）
- **仕様書の読み違いによる値の誤りを見つける唯一の完全なオラクル**。③の恒等式では捉えられない「両辺とも同じ方向に間違う」バグや、過去の修正が実は誤った仕様理解に基づいていたケースを検出できる
- 実績（2026-07-18〜19）: 第1回 1608/2337 → 2302/2337 一致。型伝播・Integer/Long オーバーフロー・Null 伝播の演算子非対称性・比較演算子の型変換・Date 演算・`DateSerial` の月日繰り上げ・`fromVbaDate` の負値丸め方向など多数のバグを発見・修正。**過去の Bug BG（Format は銀行家丸め）・Bug BK（CStr(Null)=""）が誤った仕様理解だったことも判明し訂正**（実 VBA 差分でしか見つけられない性質のバグ）。第2回は「未解明」だった `\`/`Mod` の Boolean 型保持規則・CDate の `"H.N"`/`"M,Y"` 文字列解釈を深掘りし、境界値を狙った 149 式を追加（計 2486 式）→ **2450/2486 一致**（追加分は 148/149 が初回一致し、実装した規則の正確さを裏付け）。残り 36 件は環境依存として許容登録
- **再実行タイミング**: 型強制・演算子昇格・日付計算・Format 等を変更したとき。Excel 側の再実行は Windows 環境が必要なため、`run-diff-local.ts` の結果だけを見て事前チェックする運用でも意味がある（既存の allowlist に対する新規逸脱がないか確認できる）

## 未実施の手法（候補）

### ⑥ ミューテーションテスト（Stryker 等）

- エンジンコードを機械的に改変（`<` → `<=`、`+` → `-` 等）してテストが落ちるかを確認する
- 落ちないミュータント = **その挙動を検証しているテストが存在しない**箇所。サイレントバグの温床の特定に有効
- 実行時間が非常に長い（全ミュータント × 全テスト）ため、対象を `src/engine/coerce.ts` など小さいファイルに絞るか、CI の夜間ジョブ向き

### ⑨ 文法ベースのプログラム生成ファザー

- ランダムな VBA プログラム（ネストした If/For/Select、クラス生成、ReDim、GoTo、エラーハンドラーの組み合わせ）を文法から生成して実行し、JS 生例外だけを検出する
- ①の「文・制御構文レベル」版。②を汎用化したものとも言える
- 生成器の作り込みが必要（有効なプログラムを高確率で生成しないと、パースエラーばかりで実行系に到達しない）

## 運用メモ

- どのツールも「検出 → 手動で仕様裁定（MS-VBAL / 実 VBA 通説）→ 修正 → リグレッションテスト固定」の流れ。**ツールの報告をそのままバグと断定しない**
- ツール自体の検出能力を疑うこと。③の `--selftest` のように「わざと壊して検出されるか」を確認する仕組みを持たせるのが望ましい
- 新しい修正で他ツールの前提が変わることがある（例: `Write #` の CRLF 化で既存テストの LOF 期待値が変化）。大きな修正後は関連ツールを再実行する
